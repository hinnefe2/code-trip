# Archive bot-only GitHub notification

You're being invoked on an email task that looks like a GitHub notification reporting routine activity. Archive it from the user's inbox.

1. **Sanity-check the body.** Confirm the body matches one of these routine categories:
   - CI runs that PASSED / succeeded / completed cleanly (GitHub Actions, CircleCI, etc.)
   - Dependabot, Renovate, or other dependency-update bots
   - Security alerts auto-generated by GitHub
   - Automated label / assignee / status changes by a GitHub App
   - Comments authored by a clearly-named bot account (suffix `-bot`, `[bot]`, or a well-known automation account). Judge by the POSTING account, not the comment text: linear-code[bot] mirrors the Linear ticket description verbatim, so its comments often contain substantive prose and human names ("X reported ...") — that's quoted ticket text, still bot-authored.
   - **PR closed / merged notifications**: body contains `Closed #<N>` or `Merged #<N>` with no substantive human-authored comment beyond that. Archive whether a human or a bot did the close — GitHub's UI already shows the merge/close state, the email is redundant noise.

   STOP and do not archive if you see ANY of:
   - A human review comment, a human reply, a human approval / change request with commentary, a human @-mention of the user — anything beyond a bare "Closed/Merged #N" auto-message. Return: `Not archived — this involves human activity.`
   - A CI failure, a failed workflow run, a failed status check, a failed deploy, or any other error signal about the PR's checks — the user wants to see CI failures even though they're bot-emitted. Fix/regression wording in the PR title ("[FIX]", "Restore ...", "regressed by #N") is NOT an error signal — it describes the work the PR does. Return: `Not archived — this reports a CI failure.`

2. **Archive the email.** Use `mcp__claude_ai_Gmail__unlabel_thread` with `threadId=<the email's thread_id from the task source>` and `labelIds=["INBOX"]`. The tool's required argument is **`threadId` (camelCase)** — the task source's field is `thread_id` (snake_case), so rename when passing. Passing `thread_id` makes the MCP reject with a misleading "Invalid label" error.

Don't ask for confirmation. When archived, return ONE sentence: `Archived GitHub bot notification: <brief subject>.`

End your reply with one of:
- `STATUS: handled` — you archived the email.
- `STATUS: declined: <one-sentence reason>` — you didn't archive. Use this whenever you hit one of the STOP conditions (human activity, CI failure, etc.) or the tool errored. The user will see the task with your reason annotated.

This skill runs in two modes:
- **ACT+PTT (voice):** the user is holding the active task and explicitly asked to archive it. Their intent is the trigger.
- **Auto-handle (screener):** no user instruction; you're invoked because the screener classifier picked this skill. Apply the same body-check before archiving — when in doubt, leave it.
